# /req-gate-check

需求门控检查 skill，评估需求是否满足进入下一层的条件

## Usage
/req-gate-check [需求文档路径或内容]

## Implementation
执行三个关键门控检查，确保需求质量

---

## Gate 1: Problem Confidence Gate

### 目的
评估问题定义的清晰度和完整性

### 评分标准 (0-5 分)

**0 分 - 无问题上下文**
- 只有解决方案请求
- 没有问题描述
- 示例: "我们需要一个 AI 功能"

**1 分 - 有问题但无证据**
- 有问题陈述
- 但没有证据或利益相关者
- 示例: "用户抱怨系统慢"

**2 分 - 目标不清晰**
- 有目标
- 但约束和成功指标不清晰
- 示例: "提升系统性能"

**3 分 - 基本清晰** ✅ **通过门槛**
- 目标清晰
- 成功指标明确
- 约束基本清晰
- 示例: "将 API 响应时间从 500ms 降到 200ms"

**4 分 - 清晰且有证据**
- 目标/证据/约束/利益相关者都清晰
- 边界风险已识别
- 示例: 包含当前性能数据、用户反馈、技术约束

**5 分 - 完全验证**
- 清晰 + 已验证
- 有反事实分析（如果不做会怎样）
- 有明确的非目标
- 示例: 完整的问题简报 + 验证数据

### 检查清单

- [ ] 有明确的问题陈述
- [ ] 有可测量的成功指标
- [ ] 有明确的约束条件（硬约束/软约束）
- [ ] 识别了利益相关者（决策者/否决者/影响者）
- [ ] 有当前状态的证据（数据/日志/反馈）
- [ ] 分析了"如果不做"的后果
- [ ] 明确了非目标（不做什么）
- [ ] 识别了主要风险

### 决策规则

```
if score < 3:
    ❌ 不得进入结构层/精度层/语义层
    → 补救: 提出针对性问题，收集缺失信息
else:
    ✅ 可以继续
```

### 输出格式

```markdown
## Problem Confidence Gate 评估

**评分**: X/5

**评分理由**:
- [具体原因]

**缺失信息**:
1. [缺失项 1]
2. [缺失项 2]

**建议问题**:
1. [问题 1]
2. [问题 2]

**决策**: ✅ 通过 / ❌ 不通过
```

## Gate 2: Testability Gate

### 目的
确保所有验收标准可观测、可测量、非主观

### 禁止词汇列表

**性能相关**:
- "快速"、"慢"、"高性能"、"低延迟"
- 替代: 具体的时间/吞吐量指标

**稳定性相关**:
- "稳定"、"可靠"、"健壮"
- 替代: 具体的可用性/错误率指标

**用户体验相关**:
- "用户友好"、"易用"、"直观"、"美观"
- 替代: 具体的任务完成时间/错误率

**安全相关**:
- "安全"、"足够安全"、"加密"（不具体）
- 替代: 具体的安全标准/加密算法

**质量相关**:
- "优化"、"最佳"、"合理"、"适当"
- 替代: 具体的优化目标和指标

**扩展性相关**:
- "可扩展"、"灵活"、"可配置"
- 替代: 具体的扩展点和配置项

### 检查方法

对每个验收标准，检查：

1. **可观测性**: 能否通过日志/指标/数据库查询验证？
2. **可测量性**: 有没有具体的数值/阈值？
3. **非主观性**: 不同人测试会得到相同结果吗？
4. **可自动化**: 能否编写自动化测试？

### 转换示例

| 模糊表述 | 可测量表述 |
|---------|-----------|
| ❌ 系统应该快速响应 | ✅ API 响应时间 P95 < 200ms |
| ❌ 界面应该用户友好 | ✅ 新用户完成首次下单 < 3 分钟 |
| ❌ 系统应该稳定 | ✅ 月度可用性 ≥ 99.9% |
| ❌ 数据应该安全 | ✅ 使用 AES-256 加密静态数据 |
| ❌ 性能应该优化 | ✅ 首页加载时间 < 2 秒（3G 网络） |

### 检查清单

- [ ] 所有验收标准都有具体的数值/阈值
- [ ] 没有使用禁止词汇
- [ ] 每个标准都可以通过工具验证
- [ ] 标准是客观的，非主观的
- [ ] 标准可以自动化测试

### 输出格式

```markdown
## Testability Gate 评估

**检查结果**:

### 发现的问题

1. **模糊表述**: "系统应该快速"
   - 位置: 用户故事 #3, 验收标准 2
   - 建议: 改为 "API 响应时间 P95 < 200ms"

2. **主观表述**: "界面应该美观"
   - 位置: 用户故事 #5, 验收标准 1
   - 建议: 改为 "通过 WCAG 2.1 AA 级无障碍标准"

### 统计
- 总验收标准数: X
- 可测量标准数: Y
- 需要改进: Z

**决策**: ✅ 通过 / ❌ 不通过（需改进 Z 项）
```

## Gate 3: Structure Before Precision Gate

### 目的
确保在精确化之前建立了全局结构

### 检查规则

**规则**: 没有故事地图骨架，不得创建决策表/实例表

**原因**: 避免过早精确化，先建立全局视图

### 检查项

1. **是否有故事地图骨架？**
   - Backbone（主要活动）
   - 用户旅程
   - 至少一个发布切片

2. **是否有用例或用户故事？**
   - 至少有高层次的用例
   - 或者有用户故事列表

3. **是否过早精确化？**
   - 有决策表但没有故事地图
   - 有详细实例但没有用例
   - 有 Given/When/Then 但没有用户故事

### 检查清单

- [ ] 有用户故事地图骨架（Backbone）
- [ ] 有用户旅程定义
- [ ] 有至少一个发布切片（MVP）
- [ ] 用例或用户故事已定义
- [ ] 决策表/实例表基于已有的故事

### 输出格式

```markdown
## Structure Before Precision Gate 评估

**检查结果**:

### 结构完整性
- [✅/❌] 故事地图骨架
- [✅/❌] 用户旅程
- [✅/❌] 发布切片
- [✅/❌] 用例/用户故事

### 发现的问题
- [如果有过早精确化的情况]

**决策**: ✅ 通过 / ❌ 不通过
```

## 使用场景

### 场景 1: 初始需求评估
```bash
# 在开始需求分析时
/req-gate-check initial-requirements.md

# 检查问题定义是否足够清晰
# 决定是否可以进入需求建模阶段
```

### 场景 2: 验收标准审查
```bash
# 在编写验收标准后
/req-gate-check user-stories.md

# 检查所有标准是否可测量
# 识别模糊词汇并提供改进建议
```

### 场景 3: 发布前检查
```bash
# 在交付需求文档前
/req-gate-check complete-requirements.md

# 全面检查三个门控
# 确保需求质量达标
```

## 集成到工作流

### 推荐工作流

```
1. 收集初始需求
   ↓
2. /req-gate-check → Gate 1 评估
   ↓ (如果 < 3 分)
3. 补充问题信息
   ↓
4. 重新评估 Gate 1
   ↓ (通过)
5. 创建用户故事地图
   ↓
6. /req-gate-check → Gate 3 评估
   ↓ (通过)
7. 编写验收标准
   ↓
8. /req-gate-check → Gate 2 评估
   ↓ (如果有模糊词汇)
9. 精化验收标准
   ↓
10. 最终门控检查
    ↓
11. 交付需求文档
```

## 自动化建议

### 可以自动化的检查
- 禁止词汇扫描（正则表达式）
- 数值/阈值检测
- 文档结构完整性检查

### 需要人工判断的检查
- 问题置信度评分（需要理解上下文）
- 证据充分性评估
- 利益相关者识别

---

## 质量指标

### 好的需求文档应该
- Problem Confidence Score ≥ 4
- Testability Rate ≥ 90%
- Structure Completeness = 100%

### 警告信号
- Problem Confidence Score < 3
- Testability Rate < 70%
- 有决策表但无故事地图
