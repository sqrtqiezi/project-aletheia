# /req-use-case

用例建模 skill，基于 Alistair Cockburn 的有效用例方法

## Usage
/req-use-case [功能描述或用户故事]

## Implementation
创建结构化的用例文档

### 用例模板

```markdown
## 用例: [用例名称]

### 基本信息
- **用例 ID**: UC-XXX
- **目标层次**: 用户目标层 (海平面)
- **主要参与者**: [参与者名称]
- **利益相关者**: [其他相关方]
- **前置条件**: [必须满足的条件]
- **后置条件**: [成功后的状态]

### 主成功场景
1. [参与者] [动作]
2. 系统 [响应]
3. [参与者] [动作]
4. 系统 [响应]
...

### 扩展场景
**2a. [异常情况描述]**
  2a1. 系统 [处理方式]
  2a2. [返回步骤或结束]

**3a. [变化情况描述]**
  3a1. 系统 [处理方式]
  ...

### 特殊需求
- [性能要求]
- [安全要求]
- [其他非功能需求]

### 技术和数据变化
- [可能的技术实现方式]
```

### 目标层次

根据 Cockburn 的目标层次分类：

- **Summary Level (云层)**: 多个用户目标的组合
- **User Goal Level (海平面)**: 单个用户会话的目标 ⭐ 推荐
- **Subfunction Level (鱼层)**: 支持用户目标的子功能

**建议**: 大多数用例应该在"海平面"层次

### 编写原则

1. **简洁性**: 每个步骤一句话
2. **意图而非机制**: 描述"做什么"而非"怎么做"
3. **参与者视角**: 从参与者的角度描述
4. **可验证性**: 每个步骤都可以验证
5. **完整性**: 覆盖主要场景和关键扩展

### 常见模式

#### 验证模式
```
1. 用户提交数据
2. 系统验证数据
3. 系统保存数据
4. 系统显示确认信息

扩展:
2a. 数据验证失败
  2a1. 系统显示错误信息
  2a2. 返回步骤 1
```

#### 查询模式
```
1. 用户输入查询条件
2. 系统执行查询
3. 系统显示查询结果

扩展:
2a. 无匹配结果
  2a1. 系统显示"无结果"信息
```

#### 创建模式
```
1. 用户请求创建新项目
2. 系统显示创建表单
3. 用户填写信息并提交
4. 系统验证信息
5. 系统创建项目
6. 系统显示成功信息
```

### 质量检查

- [ ] 用例名称清晰表达目标
- [ ] 主成功场景流畅完整
- [ ] 关键扩展场景已覆盖
- [ ] 前置/后置条件明确
- [ ] 步骤编号正确
- [ ] 使用领域语言
- [ ] 避免技术实现细节

### 输出
生成完整的用例文档，包括：
- 结构化的用例描述
- 主成功场景
- 扩展场景
- 前置/后置条件
- 特殊需求
