# Requirements Analyst Agent 版本对比

## 概述

本文档对比 requirements-analyst 的两个版本：
- **v1**: 基于经典理论的五层架构（纯理论版）
- **v2**: 融合经典理论与实践门控机制（理论 + 实践版）

---

## 核心差异

| 维度 | v1 (纯理论版) | v2 (理论 + 实践版) |
|------|--------------|------------------|
| **理论基础** | 五层认知架构 | 五层认知架构 + 门控机制 |
| **质量保证** | 隐式（依赖 agent 判断） | 显式（强制门控检查） |
| **可操作性** | 概念性指导 | 具体的评分标准和检查清单 |
| **失败处理** | 无明确机制 | 明确的补救措施 |
| **输出规范** | 灵活 | 严格的输出契约 |
| **可观测性** | 无 | 内置可观测性钩子 |
| **模板支持** | 无 | 内置模板 |

---

## 详细对比

### 1. 五层架构

#### v1: 纯理论描述
```markdown
第 1 层: 哲学层 - 问题本质探索
- 理论基础: Are Your Lights On?
- 目的: 确认我们在解决正确的问题
- 输出: 问题陈述、成功指标、约束条件、利益相关者
```

#### v2: 理论 + 门控机制
```markdown
第 1 层: 哲学层 - 问题本质探索
- 理论基础: Are Your Lights On?
- 目的: 确认我们在解决正确的问题
- 输出: 经过验证的问题陈述、成功指标、约束条件、利益相关者
- **门控**: problem_confidence_gate ⚠️ 可否决下层
  - 评分标准: 0-5 分
  - 通过门槛: ≥ 3 分
  - 补救措施: 提出针对性问题
```

**关键差异**: v2 增加了强制性的门控检查，确保问题定义达到最低质量标准

---

### 2. 质量保证机制

#### v1: 隐式质量保证
- 依赖 agent 的判断
- 没有明确的质量标准
- 没有强制检查点

#### v2: 显式质量门控
- **Gate 1: Problem Confidence Gate**
  - 0-5 分评分标准
  - < 3 分不得进入下层
  - 明确的补救措施

- **Gate 2: Testability Gate**
  - 禁止词汇列表（"快速"、"稳定"、"用户友好"等）
  - 强制可测量性
  - 转换示例

- **Gate 3: Structure Before Precision Gate**
  - 先建立全局结构
  - 再进行精确化
  - 避免过早优化

**关键差异**: v2 提供了可执行的质量检查机制，而不是依赖 agent 的主观判断

---

### 3. 可操作性

#### v1: 概念性指导
```markdown
## 核心能力
- 问题定义与探索
- 需求发现与分析
- 用例和故事地图建模
- 需求实例化与验证
- 领域概念建模
```

#### v2: 具体的操作指南
```markdown
## Operating Mode (操作模式)

### 默认模式: 两遍处理
**第一遍**: 探索和验证（第 1-3 层）
- 问题探索
- 质量检查
- 结构建立

**第二遍**: 规约和强化（第 4-5 层）
- 故事切片
- 实例化
- 验收标准

### 反模式（避免）
- ❌ 在验证问题陈述之前编写功能
- ❌ 在澄清规则边界之前自动化测试
- ❌ 产生详尽文档；优先最小充分制品
```

**关键差异**: v2 提供了明确的操作模式和反模式，agent 知道如何执行任务

---

### 4. 提问策略

#### v1: 无明确策略
- 没有提问限制
- 没有优先级指导

#### v2: 结构化提问策略
```markdown
## Questioning Strategy

### 限制
- 每轮最多 8 个问题
- 优先高价值问题

### 优先级顺序
1. 成功指标和可接受阈值
2. 硬约束和非目标
3. 利益相关者冲突/否决者
4. 当前状态证据
5. 边界情况和异常处理

### 提问技巧
- 重述 + 确认
- 反事实
- 边界探测
- 术语统一
```

**关键差异**: v2 限制了提问数量，提高了提问质量

---

### 5. 输出规范

#### v1: 灵活输出
- 没有明确的输出格式
- 依赖 agent 自行组织

#### v2: 严格的输出契约
```markdown
## Output Contract

### 必需章节（按顺序）
1. 执行摘要
2. 经过验证的问题陈述（第 1 层）
3. 开放问题 + 明确假设
4. 用户故事地图（第 3 层）
5. 用户故事（第 4 层）
6. 业务规则（第 5 层）
7. 实例表格（第 5 层）
8. 验收标准（第 5 层）
9. 可观测性钩子
10. 风险和失败模式
```

**关键差异**: v2 提供了标准化的输出格式，确保一致性

---

### 6. 可观测性

#### v1: 无可观测性支持
- 没有提到如何验证需求
- 没有可观测性指导

#### v2: 内置可观测性钩子
```markdown
## 可观测性钩子
- 指标: [具体的业务指标]
- 日志: [关键事件日志]
- 事件: [领域事件]
- 数据库状态: [关键表和字段]
```

**关键差异**: v2 强制要求每个需求都有可观测性钩子

---

### 7. 模板支持

#### v1: 无内置模板
- 需要 agent 自行创建格式

#### v2: 内置模板
- **问题简报模板**
- **故事地图骨架模板**
- **Given/When/Then 模板**

**关键差异**: v2 提供了开箱即用的模板，降低使用门槛

---

### 8. 质量标准

#### v1: 无明确标准
- 没有定义"好的需求"

#### v2: 明确的质量标准
```markdown
## Quality Bar

### 必须（Must）
- ✅ 在第 1 层满足之前，不提出解决方案
- ✅ 每个故事必须映射到可测量的结果或规则
- ✅ 每个规则必须至少有 2 个实例，包括 1 个边界/反例
- ✅ 验收标准必须可自动化和可观测

### 应该（Should）
- 当规则基于多个条件分支时，使用决策表
- 提供最小发布切片（薄垂直切片）
```

**关键差异**: v2 提供了可检查的质量标准

---

## 新增的 Skill

### v2 新增: /req-gate-check

专门用于执行门控检查的 skill：

**功能**:
- Gate 1: Problem Confidence Gate 评估（0-5 分评分）
- Gate 2: Testability Gate 评估（禁止词汇扫描）
- Gate 3: Structure Before Precision Gate 评估（结构完整性检查）

**输出**:
- 综合评估报告
- 具体的改进建议
- 行动计划

## 使用场景对比

### 场景 1: 初始需求收集

#### v1 工作流
```
1. 用户提供需求
2. Agent 分析需求
3. Agent 输出需求文档
```

#### v2 工作流
```
1. 用户提供需求
2. Agent 执行 Gate 1 评估
3. 如果 < 3 分，提出针对性问题
4. 用户补充信息
5. 重新评估 Gate 1
6. 通过后继续下一层
```

**差异**: v2 有明确的质量检查点，确保不会在问题不清晰时继续

---

### 场景 2: 编写验收标准

#### v1 工作流
```
1. Agent 编写验收标准
2. 用户审查
```

#### v2 工作流
```
1. Agent 编写验收标准
2. Agent 执行 Gate 2 评估
3. 扫描禁止词汇
4. 如果发现模糊表述，自动提供改进建议
5. Agent 修正验收标准
6. 用户审查
```

**差异**: v2 自动检测和修正模糊表述

## 理论完整性对比

### v1: 完整的理论覆盖
- 五层架构完整
- 理论基础清晰
- 适合学习和理解

### v2: 理论 + 实践平衡
- 保留五层架构
- 增加实践门控
- 适合实际项目使用

**结论**: v2 不是替代 v1，而是在 v1 的基础上增加了实践可操作性

---

## 推荐使用场景

### 使用 v1 的场景
- 学习需求工程理论
- 理解五层认知架构
- 需要灵活的需求分析方法
- 团队有经验的需求分析师

### 使用 v2 的场景
- 实际项目中的需求分析
- 需要强制质量保证
- 团队缺乏需求分析经验
- 需要标准化的输出格式
- 需要可观测性和可测量性

---

## 迁移指南

### 从 v1 迁移到 v2

**步骤 1: 理解门控机制**
- 阅读三个门控的评分标准
- 理解通过门槛和补救措施

**步骤 2: 采用输出契约**
- 使用 v2 的标准输出格式
- 使用内置模板

**步骤 3: 集成门控检查**
- 在关键节点使用 /req-gate-check
- 根据评估结果调整需求

**步骤 4: 建立质量标准**
- 采用 v2 的质量标准
- 定期检查需求质量
