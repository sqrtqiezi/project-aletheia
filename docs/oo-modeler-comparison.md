# OO Modeler Agent 版本对比

## 概述

本文档对比 oo-modeler 的两个版本：
- **v1**: 基于经典理论的四层架构（纯理论版）
- **v2**: 融合经典理论与设计门控机制（理论 + 实践版）

---

## 核心差异

| 维度 | v1 (纯理论版) | v2 (理论 + 实践版) |
|------|--------------|------------------|
| **理论基础** | 四层认知架构 | 四层认知架构 + 门控机制 |
| **质量保证** | 隐式（依赖 agent 判断） | 显式（强制门控检查） |
| **反模式检测** | 无 | 5 种常见反模式自动检测 |
| **输出规范** | 灵活 | 严格的输出契约（10 个必需章节） |
| **设计决策** | 无记录 | 内置设计决策记录模板 |
| **耦合控制** | 原则性指导 | 具体的耦合度阈值 |

---

## 详细对比

### 1. 四层架构

#### v1: 纯理论描述
```markdown
第 1 层: 责任驱动核心
- 理论基础: Object Design (Wirfs-Brock)
- 目的: 职责分配与协作设计
```

#### v2: 理论 + 门控机制
```markdown
第 1 层: 责任驱动核心
- 理论基础: Object Design (Wirfs-Brock)
- 目的: 识别对象并分配职责
- **门控**: responsibility_clarity_gate ⚠️ 可否决下层
  - 评分标准: 0-5 分
  - 通过门槛: ≥ 3 分
  - 检查: 上帝对象、懒惰对象、职责清晰度
```

**关键差异**: v2 增加了可量化的职责清晰度评估

---

### 2. 质量保证机制

#### v1: 原则性指导
```markdown
## Working Principles
1. 职责优先于数据结构
2. 场景驱动设计
3. 领域语言为中心
4. 边界清晰
5. 协作简单
6. 充血模型
```

这些是好的原则，但缺乏具体的检查标准。

#### v2: 显式质量门控
- **Gate 1: Responsibility Clarity Gate**
  - 0-5 分评分标准
  - 检查清单：上帝对象、懒惰对象、职责清晰度
  - < 3 分不得进入下层

- **Gate 2: Coupling Control Gate**
  - 禁止模式：循环依赖、过度耦合、双向关联
  - 具体阈值：依赖数 ≤ 5，被依赖数 ≤ 10
  - 依赖图分析

- **Gate 3: Domain Alignment Gate**
  - 统一语言对齐检查
  - Entity/Value Object 分类验证
  - 聚合边界合理性检查

**关键差异**: v2 提供了可执行的检查标准和具体阈值

---

### 3. 反模式检测

#### v1: 无反模式检测
- 依赖 agent 的经验
- 没有系统化的检测机制

#### v2: 5 种反模式自动检测

**1. God Object (上帝对象)**
```
检测规则:
- 职责数 > 5
- 方法数 > 20
- 被依赖数 > 10
```

**2. Anemic Domain Model (贫血模型)**
```
检测规则:
- 只有 getter/setter
- Doing 职责 = 0
- 业务逻辑在外部 Service
```

**3. Circular Dependency (循环依赖)**
```
检测规则:
- 依赖图中存在环
- DFS 检测 back edge
```

**4. Inappropriate Intimacy (过度亲密)**
```
检测规则:
- 双向关联
- 直接访问内部状态
```

**5. Feature Envy (特性依恋)**
```
检测规则:
- 方法大量使用另一个对象的数据
- 职责分配错误
```

**关键差异**: v2 提供了系统化的反模式检测机制

---

### 4. 输出规范

#### v1: 灵活输出
```markdown
## Output Formats
- CRC 卡片
- 职责分配矩阵
- UML 类图 (PlantUML)
- UML 序列图 (PlantUML)
- 聚合设计文档
- 领域服务定义
```

没有明确的章节顺序和必需内容。

#### v2: 严格的输出契约
```markdown
## Output Contract

### 必需章节（按顺序）
1. 执行摘要
2. 对象清单（CRC 卡片）
3. 职责分配矩阵（GRASP 模式）
4. 交互序列（序列图）
5. 聚合设计（DDD 战术模式）
6. 领域服务
7. 类图（PlantUML）
8. 设计决策记录
9. 质量评估（门控评分）
10. 风险和改进建议
```

**关键差异**: v2 提供了标准化的输出格式，确保一致性

---

### 5. 设计决策记录

#### v1: 无设计决策记录
- 设计决策隐含在设计中
- 没有记录决策理由和权衡

#### v2: 内置设计决策记录模板
```markdown
## 设计决策: [决策标题]

**上下文**: [什么情况下需要做这个决策]

**选项**:
1. 选项 A: [优点/缺点]
2. 选项 B: [优点/缺点]

**决策**: [选择了哪个选项]

**理由**: [为什么选择这个选项]

**后果**: [这个决策的影响]
```

**关键差异**: v2 强制记录设计决策，提高可维护性

---

### 6. 耦合控制

#### v1: 原则性指导
```markdown
## Working Principles
5. 协作简单
```

没有具体的耦合度标准。

#### v2: 具体的耦合度阈值
```markdown
## Gate 2: Coupling Control Gate

**具体阈值**:
- 每个对象的依赖数（出度）≤ 5
- 每个对象的被依赖数（入度）≤ 10
- 平均依赖数 ≤ 3
- 无循环依赖

**禁止模式**:
- ❌ 循环依赖
- ❌ 过度耦合
- ❌ 双向关联（除非必要）
```

**关键差异**: v2 提供了可量化的耦合度标准

---

### 7. 领域对齐

#### v1: 概念性指导
```markdown
## Working Principles
3. 领域语言为中心
```

#### v2: 系统化的对齐检查
```markdown
## Gate 3: Domain Alignment Gate

**检查项**:
- [ ] 对象命名使用统一语言词汇表
- [ ] Entity/Value Object 分类正确
- [ ] 聚合边界清晰
- [ ] 领域服务职责合理
- [ ] Repository 只为聚合根创建

**技术术语检测**:
禁止: Manager, Service, Handler, Processor, Helper, Util, DAO, DTO
```

**关键差异**: v2 提供了具体的检查清单和禁止词汇

---

### 8. 操作模式

#### v1: 无明确操作模式
- 依赖 agent 自行决定工作流程

#### v2: 两遍处理模式
```markdown
## Operating Mode

### 第一遍: 对象识别与职责分配（第 1-2 层）
- 从用例识别候选对象
- 创建 CRC 卡片
- 设计对象交互
- 应用 GRASP 模式

### 第二遍: 领域建模与形式化（第 3-4 层）
- 应用 DDD 战术模式
- 识别聚合边界
- 创建 UML 图
- 验证设计质量

### 反模式（避免）
- ❌ 数据驱动设计
- ❌ 过早形式化
- ❌ 贫血模型
- ❌ 过度使用继承
- ❌ 忽略领域语言
```

**关键差异**: v2 提供了明确的工作流程和反模式警告

---

## 新增的 Skill

### v2 新增: /oo-gate-check

专门用于执行设计门控检查的 skill：

**功能**:
- Gate 1: Responsibility Clarity Gate 评估（职责清晰度评分）
- Gate 2: Coupling Control Gate 评估（耦合度分析、循环依赖检测）
- Gate 3: Domain Alignment Gate 评估（领域对齐检查）
- 5 种设计反模式自动检测

**输出**:
- 综合评估报告
- 反模式检测结果
- 具体的改进建议
- 行动计划

## 使用场景对比

### 场景 1: 对象识别

#### v1 工作流
```
1. 用户提供用例
2. Agent 识别对象
3. Agent 输出 CRC 卡片
```

#### v2 工作流
```
1. 用户提供用例
2. Agent 识别对象并创建 CRC 卡片
3. Agent 执行 Gate 1 评估
4. 如果发现上帝对象或懒惰对象，重新分配职责
5. 通过后继续下一层
```

**差异**: v2 有明确的质量检查点

---

### 场景 2: 聚合设计

#### v1 工作流
```
1. Agent 应用 DDD 战术模式
2. Agent 输出聚合设计
3. 用户审查
```

#### v2 工作流
```
1. Agent 应用 DDD 战术模式
2. Agent 执行 Gate 3 评估
3. 检查统一语言对齐
4. 检查 Entity/Value Object 分类
5. 检查聚合边界合理性
6. 如果发现问题，自动提供改进建议
7. 用户审查
```

**差异**: v2 自动检测领域对齐问题

---

### 场景 3: 设计评审

#### v1 工作流
```
1. Agent 完成设计
2. 用户手动审查
3. 发现问题后反馈
```

#### v2 工作流
```
1. Agent 完成设计
2. Agent 自动执行三个门控检查
3. Agent 自动检测 5 种反模式
4. Agent 输出质量评估报告
5. 用户审查（已有明确的问题列表）
```

**差异**: v2 提供了自动化的质量评估

---

## 理论完整性对比

### v1: 完整的理论覆盖
- 四层架构完整
- 理论基础清晰
- 适合学习和理解

### v2: 理论 + 实践平衡
- 保留四层架构
- 增加实践门控
- 适合实际项目使用

**结论**: v2 不是替代 v1，而是在 v1 的基础上增加了实践可操作性

---

## 推荐使用场景

### 使用 v1 的场景
- 学习面向对象设计理论
- 理解四层认知架构
- 需要灵活的设计方法
- 团队有经验的架构师

### 使用 v2 的场景
- 实际项目中的对象设计
- 需要强制质量保证
- 团队缺乏设计经验
- 需要标准化的输出格式
- 需要反模式检测
- 需要设计决策记录

## 迁移指南

### 从 v1 迁移到 v2

**步骤 1: 理解门控机制**
- 阅读三个门控的评分标准
- 理解通过门槛和补救措施
- 学习 5 种反模式

**步骤 2: 采用输出契约**
- 使用 v2 的标准输出格式
- 使用内置模板（CRC、聚合、设计决策）

**步骤 3: 集成门控检查**
- 在关键节点使用 /oo-gate-check
- 根据评估结果调整设计

**步骤 4: 建立质量标准**
- 采用 v2 的质量标准
- 定期检查设计质量

## 与 requirements-analyst-v2 的协同

### 统一的门控理念

**requirements-analyst-v2 的门控**:
1. Problem Confidence Gate（问题置信度）
2. Testability Gate（可测试性）
3. Structure Before Precision Gate（结构优先）

**oo-modeler-v2 的门控**:
1. Responsibility Clarity Gate（职责清晰度）
2. Coupling Control Gate（耦合控制）
3. Domain Alignment Gate（领域对齐）

### 工作流集成

```
requirements-analyst-v2
    ↓ 输出
统一语言词汇表 + 领域概念模型
    ↓ DDD 桥梁
oo-modeler-v2
    ↓ 输出
对象模型 + 战术模式
    ↓
实现阶段
```

**关键点**:
- requirements-analyst-v2 的 Gate 3（结构优先）确保有全局视图
- oo-modeler-v2 的 Gate 3（领域对齐）确保设计与需求一致
- 两者通过 DDD 桥梁无缝连接

## 总结

| 特性 | v1 | v2 |
|------|----|----|
| **理论完整性** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| **实践可操作性** | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ |

| **反模式检测** | ⭐ | ⭐⭐⭐⭐⭐ |

**推荐**:
- 如果你是 OO 设计初学者，从 v1 开始理解理论
- 如果你在实际项目中使用，选择 v2
- 如果你的团队需要标准化，选择 v2
- 如果你需要灵活性，选择 v1
