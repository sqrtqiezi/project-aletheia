# /req-example

需求实例化 skill，基于 Gojko Adzic 的实例化需求方法 (Specification by Example)

## Usage
/req-example [用户故事或需求描述]

## Implementation
将抽象需求转化为具体的、可验证的实例

### Given-When-Then 格式

最常用的实例化格式，源自 BDD (Behavior-Driven Development)

```gherkin
Feature: [功能名称]
  [功能描述]

Scenario: [场景名称]
  Given [前置条件]
  When [触发动作]
  Then [预期结果]
```

### 实例化步骤

#### 1. 从用户故事开始

**用户故事**:
```
作为购买者，我想要在购物车中应用优惠券，以便享受折扣
```

#### 2. 识别关键实例

通过提问识别关键实例：
- 正常情况是什么？
- 边界情况是什么？
- 异常情况是什么？

**关键实例**:
- 有效优惠券，满足使用条件
- 有效优惠券，但不满足最低金额
- 优惠券已过期
- 优惠券已被使用
- 优惠券不存在

#### 3. 编写具体实例

**实例 1: 成功应用优惠券**
```gherkin
Scenario: 成功应用满减优惠券
  Given 购物车中有商品，总金额为 150 元
  And 我有一张"满 100 减 20"的优惠券，有效期至 2026-12-31
  When 我在购物车中输入优惠券代码 "SAVE20"
  Then 优惠券应用成功
  And 订单金额应该是 130 元
  And 显示"已优惠 20 元"
```

**实例 2: 不满足最低金额**
```gherkin
Scenario: 优惠券不满足最低使用金额
  Given 购物车中有商品，总金额为 80 元
  And 我有一张"满 100 减 20"的优惠券
  When 我在购物车中输入优惠券代码 "SAVE20"
  Then 优惠券应用失败
  And 显示错误信息"订单金额不满足优惠券使用条件（需满 100 元）"
  And 订单金额保持 80 元
```

**实例 3: 优惠券已过期**
```gherkin
Scenario: 使用已过期的优惠券
  Given 购物车中有商品，总金额为 150 元
  And 我有一张"满 100 减 20"的优惠券，有效期至 2025-12-31
  And 当前日期是 2026-01-01
  When 我在购物车中输入优惠券代码 "SAVE20"
  Then 优惠券应用失败
  And 显示错误信息"优惠券已过期"
  And 订单金额保持 150 元
```

### 实例表格

对于数据驱动的场景，使用表格更清晰：

```gherkin
Scenario Outline: 计算不同优惠券的折扣
  Given 购物车总金额为 <原价> 元
  When 应用优惠券 "<优惠券类型>"
  Then 最终金额应该是 <最终价> 元

  Examples:
    | 原价 | 优惠券类型    | 最终价 |
    | 150  | 满100减20    | 130    |
    | 150  | 9折券        | 135    |
    | 150  | 满200减50    | 150    |
    | 80   | 满100减20    | 80     |
```

### 实例化模板

```markdown
## 功能: [功能名称]

### 用户故事
作为 [角色]，我想要 [功能]，以便 [价值]

### 验收标准

#### 场景 1: [正常场景名称]
**Given** (前置条件):
- [具体的初始状态]
- [相关的数据]

**When** (触发动作):
- [用户执行的具体操作]

**Then** (预期结果):
- [系统的响应]
- [状态的变化]
- [用户看到的反馈]

---

#### 场景 2: [边界场景名称]
[重复上述结构]

---

#### 场景 3: [异常场景名称]
[重复上述结构]

---

### 业务规则
- [规则 1]
- [规则 2]

### 数据示例
[提供具体的测试数据]
```

### 实例化原则

1. **具体性**: 使用具体的数据，而非抽象描述
   - ❌ "用户输入有效的优惠券"
   - ✅ "用户输入优惠券代码 'SAVE20'"

2. **可验证性**: 每个实例都可以自动化验证
   - ❌ "系统正确处理"
   - ✅ "订单金额应该是 130 元"

3. **完整性**: 覆盖正常、边界、异常情况
   - 正常路径 (Happy Path)
   - 边界条件 (Boundary)
   - 异常情况 (Exception)

4. **简洁性**: 每个实例专注于一个测试点
   - 避免在一个场景中测试多个功能

5. **可读性**: 业务人员和技术人员都能理解
   - 使用领域语言
   - 避免技术术语

### 关键实例类型

#### 正常实例 (Happy Path)
最常见的、预期的使用场景

#### 边界实例 (Boundary)
- 最小值、最大值
- 空值、零值
- 临界点

#### 异常实例 (Exception)
- 无效输入
- 权限不足
- 资源不可用
- 超时

#### 组合实例 (Combination)
多个条件的组合

### 协作式规约

实例化是团队协作的过程：

**三方会议 (Three Amigos)**:
- 业务代表 (Product Owner)
- 开发人员 (Developer)
- 测试人员 (Tester)

**会议流程**:
1. 业务代表讲解用户故事
2. 团队提问澄清需求
3. 共同识别关键实例
4. 编写 Given-When-Then 场景
5. 确认实例覆盖完整

### 活文档

实例化的需求可以：
- 作为验收测试自动执行
- 作为活文档持续更新
- 作为回归测试保护变更

### 质量检查

- [ ] 使用具体的数据和值
- [ ] 每个场景可以独立验证
- [ ] 覆盖正常、边界、异常情况
- [ ] 使用领域语言
- [ ] Given-When-Then 结构清晰
- [ ] 避免技术实现细节
- [ ] 业务人员可以理解

### 输出
生成实例化需求文档，包括：
- Given-When-Then 场景
- 实例表格（如适用）
- 业务规则说明
- 测试数据示例
